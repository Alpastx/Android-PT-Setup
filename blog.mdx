# Automating an Android Penetration Testing Lab from Scratch

In this post, we build a **fully automated** Android penetration testing environment from scratch — no Android Studio, no manual clicking, no guesswork. We download the SDK directly from Google, create two Android Virtual Devices: **A10** gets a writable system, Burp CA in the system store, and an HTTP proxy to your host; **A14PR** is rooted with **Magisk** via rootAVD. Host pentesting tools install via **pipx**. All with a single command. We also explore the internals of how each piece works so you can adapt it to your own workflow.

---

## Why Automate?

If you've ever set up an Android pentesting lab manually, you know the pain:

- Downloading the SDK, organizing folders, setting `ANDROID_HOME`
- Creating AVDs, patching `config.ini` for hardware keys
- Converting certificates, disabling AVB, remounting partitions
- Rooting with Magisk, praying `adb remount` doesn't fail silently
- Doing it all over again when something breaks

Every step has a timing dependency. Run `adb remount` before the system is fully booted? Silent failure. Push a certificate before getting root? Nothing happens. Forget `hw.keyboard=yes`? Enjoy tapping on-screen keys forever.

**Android-PT-Setup** automates this into a single `bash setup.sh` that takes you from zero to a working **A10** (Burp + proxy) and **A14PR** (Magisk).

By the end of this setup, you'll have:

| Device | Android Version | Root Method | Key Features |
|--------|----------------|-------------|-------------|
| **A10** | Android 10 (API 29) | `adb root` + writable system + AVB disabled | Burp CA in system store, HTTP proxy to host (`10.0.2.2`) |
| **A14PR** | Android 14 (API 34) | Magisk (systemless, via rootAVD) | Play Store; no automated Burp/proxy in this repo |

## The Shared Library: `lib.sh`

Before diving into the individual scripts, let's look at the foundation. Every script sources `lib.sh`, which provides cross-distro support and — most importantly — **reliable emulator boot detection**.

### The Boot Detection Problem

The single biggest source of failed rooting attempts is **timing**. The naive approach looks like this:

```bash
# The classic approach that fails 40% of the time
adb wait-for-device
sleep 10          # Is 10 seconds enough? Maybe. Maybe not.
adb root
sleep 5           # Hope the device is ready...
adb remount       # Silent failure if system isn't mounted yet
```

`adb wait-for-device` returns as soon as the ADB daemon responds — **not** when Android is fully booted. The system partition might still be mounting. SELinux might not be initialized. Your `adb remount` silently fails on a read-only filesystem, and you push your certificate into the void.

The fix is to **poll the actual boot state**:

```bash
wait_for_boot() {
    local timeout="${1:-120}"
    local elapsed=0
    local interval=3

    log_info "Waiting for device to fully boot (timeout: ${timeout}s)..."
    adb wait-for-device 2>/dev/null

    while (( elapsed < timeout )); do
        local boot_status
        boot_status=$(adb shell getprop sys.boot_completed 2>/dev/null | tr -d '\r\n' || true)
        if [[ "$boot_status" == "1" ]]; then
            log_ok "Device booted (${elapsed}s)"
            return 0
        fi

        # Check emulator process is still alive
        if [[ -n "$EMULATOR_PID" ]] && ! kill -0 "$EMULATOR_PID" 2>/dev/null; then
            log_warn "Emulator process (PID $EMULATOR_PID) died"
            return 1
        fi

        sleep "$interval"
        (( elapsed += interval ))
    done

    log_warn "Boot timeout after ${timeout}s"
    return 1
}
```

This function:
1. Waits for ADB to see the device
2. Polls `sys.boot_completed` every 3 seconds
3. Detects if the emulator process crashed (PID tracking)
4. Times out after a configurable duration (default 120s)

No more guessing with arbitrary sleeps.

### Cross-Distro Package Management

The library also handles package installation across distros:

```bash
pkg_install() {
    local distro
    distro=$(detect_distro)

    local -a resolved=()
    for pkg in "$@"; do
        case "$pkg" in
            jdk)
                case "$distro" in
                    arch)           resolved+=("jdk") ;;
                    debian|ubuntu)  resolved+=("openjdk-17-jdk") ;;
                    fedora)         resolved+=("java-17-openjdk") ;;
                esac ;;
            pipx)
                case "$distro" in
                    arch)           resolved+=("python-pipx") ;;
                    debian|ubuntu)  resolved+=("pipx") ;;
                    fedora)         resolved+=("pipx") ;;
                esac ;;
            # openssl, git, unzip, curl — same name everywhere
            *)  resolved+=("$pkg") ;;
        esac
    done

    case "$distro" in
        arch)           yay -S --noconfirm --needed "${resolved[@]}" ;;
        debian|ubuntu)  sudo apt-get install -y "${resolved[@]}" ;;
        fedora)         sudo dnf install -y "${resolved[@]}" ;;
    esac
}
```

So in any script, you just write `pkg_install jdk pipx openssl` and it does the right thing on Arch, Ubuntu, or Fedora.

## Step 1: Downloading the SDK Tools

`Tools-downloader.sh` grabs three things from official sources:

1. **Platform tools** (adb, fastboot) — from `dl.google.com`
2. **Command-line tools** (sdkmanager, avdmanager) — URL extracted dynamically from the Android Studio download page
3. **Magisk** — latest APK from GitHub releases API

```bash
# Platform tools — straightforward download
PlatformToolsL="https://dl.google.com/android/repository/platform-tools-latest-linux.zip"

# Cmdline tools — we scrape the actual download URL from Google's page
url=$(curl -fsSL "https://developer.android.com/studio#cmdline-tools" | \
  grep -oP 'https://dl.google.com/android/repository/commandlinetools-linux-[0-9]+_latest\.zip' | \
  head -n 1)

# Magisk — latest release from GitHub API
release_json=$(curl -fsSL "https://api.github.com/repos/topjohnwu/Magisk/releases/latest")
apk_url=$(echo "$release_json" | \
  grep -oP '"browser_download_url":\s*"\K[^"]+Magisk-v[^"]+\.apk' | \
  head -n 1)
```

Each download is cached — if the file already exists, it skips. This means you can re-run `setup.sh` without re-downloading everything.

> **Note on Magisk naming**: The file downloads as an APK but gets saved as `Magisk.zip`. This is intentional — APK files are valid ZIPs, and rootAVD expects `Magisk.zip` as input.

## Step 3: Installing System Images and Creating AVDs

With `sdkmanager` now in PATH, we install two system images:

```bash
# Android 10 — google_apis (no Play Store, easier to root via system modification)
sdkmanager --install "system-images;android-29;google_apis;x86"

# Android 14 — google_apis_playstore (includes Play Store, rooted via Magisk)
sdkmanager --install "system-images;android-34;google_apis_playstore;x86_64"
```

**Why two different image types?**

- `google_apis` (A10): No Play Store, but the system partition can be modified directly. Perfect for installing CA certificates into `/system/etc/security/cacerts/`.
- `google_apis_playstore` (A14PR): Includes Google Play Store for testing real-world app behavior. System partition is protected, so we use Magisk for systemless root.

Then we create the AVDs:

```bash
avdmanager create avd -n "A10" -k "system-images;android-29;google_apis;x86" --force
avdmanager create avd -n "A14PR" -k "system-images;android-34;google_apis_playstore;x86_64" --force
```

### Fixing Hardware Keys

Fresh AVDs have broken hardware buttons by default. `HWKeys.sh` patches each AVD's `config.ini`:

```bash
# The script searches both possible AVD config locations
for dir in "$HOME/.android/avd" "$HOME/.config/.android/avd"; do
    if [[ -f "$dir/${AVD_NAME}.avd/config.ini" ]]; then
        ConfigFile="$dir/${AVD_NAME}.avd/config.ini"
        break
    fi
done

sed -i '/^hw\.keyboard=/d' "$ConfigFile"
sed -i '/^hw\.mainKeys=/d' "$ConfigFile"
echo "hw.keyboard=yes" >> "$ConfigFile"
echo "hw.mainKeys=yes" >> "$ConfigFile"
```

Without this, you're stuck tapping the on-screen keyboard for every command.

## Step 4: Rooting A10 and Installing the Burp Certificate

This is where most manual setups fail. The A10 rooting process requires a precise sequence with proper timing between each step. Here's how `rootAVD.sh` handles it:

### Certificate Preparation

First, we convert the Burp certificate from DER to PEM format and generate the hash Android uses for CA certificate filenames:

```bash
# Convert DER to PEM
openssl x509 -inform der -in burp.der -out burp.cer

# Generate the hash Android uses for cert filenames
hash=$(openssl x509 -inform PEM -subject_hash_old -in burp.cer | head -1)

# Validate the hash is actually hex
if [[ -z "$hash" || ! "$hash" =~ ^[a-f0-9]+$ ]]; then
    log_fatal "Failed to extract valid certificate hash"
fi

# Android expects certs named as <hash>.0
cp burp.cer "${hash}.0"
```

Android's system certificate store at `/system/etc/security/cacerts/` expects certificates named `<subject_hash>.0`. The `-subject_hash_old` flag generates the hash format Android actually uses (OpenSSL's old algorithm).

### The Rooting Sequence

```bash
# 1. Start emulator with writable system and WAIT FOR FULL BOOT
start_emulator "A10" "-writable-system"

# 2. Get root access (with retry logic)
adb_root_and_wait

# 3. Disable Android Verified Boot
adb shell avbctl disable-verification

# 4. Reboot and wait for FULL boot (not just adb connection)
adb reboot
wait_for_boot 120

# 5. Root again after reboot
adb_root_and_wait

# 6. Remount system as read-write
adb remount

# 7. Push certificate and set permissions
adb push "${hash}.0" /system/etc/security/cacerts/
adb shell chmod 644 "/system/etc/security/cacerts/${hash}.0"

# 8. Verify it's actually there
adb shell "test -f /system/etc/security/cacerts/${hash}.0"

# 9. Final reboot to apply
adb reboot
```

Every single step has error checking. Every boot wait uses `sys.boot_completed` polling instead of arbitrary sleeps. The emulator PID is tracked so we detect crashes instantly. And if anything fails, a trap handler kills the emulator and cleans up temp files.

**Why does this work?**

1. `-writable-system` tells the emulator to allow system partition modifications
2. `avbctl disable-verification` disables Android Verified Boot, which would otherwise reject our modified system partition on next boot
3. `adb remount` remounts `/system` as read-write so we can push files
4. The certificate goes into `/system/etc/security/cacerts/` — the **system** CA store, not the user store. This means apps that only trust system CAs will trust our Burp certificate

## Step 5: Rooting A14PR with Magisk

Android 14 with Play Store images protects the system partition more aggressively. Instead of modifying the system directly, we use **Magisk** for systemless root — it patches the `ramdisk.img` to inject root capabilities without touching `/system`.

We use [rootAVD](https://gitlab.com/newbit/rootAVD) — a tool specifically designed to root Android Virtual Devices with Magisk.

```bash
A14PR() {
    local android_home
    android_home=$(get_android_home)
    local rootavd_dir="$android_home/rootAVD"
    local system_image="$android_home/system-images/android-34/google_apis_playstore/x86_64/ramdisk.img"

    # Start emulator and wait for FULL boot
    start_emulator "A14PR"

    # Run rootAVD to patch ramdisk with Magisk
    (
        cd "$rootavd_dir"
        ./rootAVD.sh ../system-images/android-34/google_apis_playstore/x86_64/ramdisk.img
    )

    # Wait for device to come back after rootAVD does its thing
    wait_for_boot 120
}
```

**How rootAVD works under the hood:**

1. It takes the path to the AVD's `ramdisk.img`
2. Unpacks the ramdisk
3. Injects Magisk's init scripts and binaries
4. Repacks the ramdisk
5. On next boot, Magisk initializes before the Android framework, providing root

After this, you'll have the Magisk app available in the system. You can install Magisk modules (like a certificate installer module for A14PR if needed).

> **Path gotcha**: rootAVD expects the ramdisk path relative to `$ANDROID_HOME`. Since we `cd` into the rootAVD directory, we use `../system-images/...` to go up one level to the SDK root. Getting this path wrong is one of the most common failure modes.

## Step 7: Frida on the host vs frida-server on the device

`frida-tools` is installed on the **host** with pipx. To instrument apps on an emulator you still need **frida-server** on the **device**, matching your host `frida --version`. This project **does not** download or push frida-server automatically.

Typical workflow:

1. Download the matching `frida-server-*-android-*.xz` from [Frida releases](https://github.com/frida/frida/releases) for your emulator ABI (`adb shell getprop ro.product.cpu.abi`).
2. Extract, `adb push` to `/data/local/tmp/frida-server`, `chmod 755`.
3. Run `adb shell "/data/local/tmp/frida-server &"` (or use `adb shell` and run in background).
4. From the host: `frida-ps -U`, `frida -U -f com.example.app -l your-script.js`.

Use your own or community Frida scripts for SSL pinning bypass and root checks — they are not bundled in this repository.

## Step 8: Auto-Configuring the Burp Proxy

Instead of manually navigating Settings > Wi-Fi > Proxy on each emulator, the setup configures the proxy automatically:

```bash
configure_proxy() {
    local port="${1:-${BURP_PORT:-8080}}"
    local host="10.0.2.2"  # Android emulator alias for host's 127.0.0.1

    adb shell settings put global http_proxy "${host}:${port}"
}
```

**Why `10.0.2.2`?** Android emulators use a virtual network where `10.0.2.2` always maps to the host machine's loopback. No need to detect your actual IP.

The proxy is configured during setup. If you need to change the port:

```bash
BURP_PORT=9090 bash setup.sh
```

Or change it later on a running emulator:

```bash
adb shell settings put global http_proxy 10.0.2.2:9090
```

To remove the proxy:

```bash
adb shell settings delete global http_proxy
```

## Reliability: What Makes This Different

The key differentiator from manual setup or basic scripts is the **reliability layer** in `lib.sh`. Here's a summary of what it handles:

| Problem | Manual Approach | Our Approach |
|---------|----------------|--------------|
| Emulator not fully booted | `sleep 10` and hope | Poll `sys.boot_completed` with 120s timeout |
| Emulator crashes mid-setup | Commands hang forever | PID tracking + crash detection |
| `adb root` fails silently | Don't notice until later | Retry up to 3 times with boot verification |
| Temp files left after failure | Manual cleanup | `trap cleanup EXIT` |
| Wrong distro packages | Edit script manually | Auto-detection: Arch/Ubuntu/Fedora |
| Wrong shell RC file | Hardcoded `.zshrc` | Auto-detect from `$SHELL` |
| Re-running setup duplicates config | Double PATH entries | `grep` check before appending |
| frida client/server mismatch | Manual version matching | You choose matching frida-server build (not auto-pushed) |
| No proxy on A10 | Manual Settings navigation | Auto-configure via `adb shell settings put` on A10 |
| Burp trust on A14PR | Manual user CA / module | Not automated here — Magisk root only |

## Troubleshooting

**Emulator fails to start**
- Check you have enough disk space (`df -h`)
- Verify KVM is available: `ls /dev/kvm` (needed for hardware acceleration)
- On some systems: `sudo usermod -aG kvm $USER` and re-login

**`adb remount` fails on A10**
- The emulator MUST be started with `-writable-system`
- AVB must be disabled first (`avbctl disable-verification` + reboot)
- The device must be fully booted (not just past the ADB daemon startup)

**rootAVD fails for A14PR**
- Ensure `Magisk.zip` exists in `~/android_sdk/rootAVD/`
- Ensure the system image is installed: `ls ~/android_sdk/system-images/android-34/`
- Check the ramdisk path matches your architecture

**Frida can't connect**
- Install and run **frida-server** on the device yourself (same major/minor as host `frida --version`)
- Ensure the emulator is running and `adb devices` shows it
